    SPEC: CLAUDE.md Optimization & Command Reform v3.0¶

    Status: draft Created: 2026-02-04 From Brainstorm: BRAINSTORM-claude-md-optimization-2026-02-04.md Target Version: v2.12.0

    Overview¶

    Redesign the CLAUDE.md command suite to enforce token-efficient CLAUDE.md files using a pointer/trigger architecture. Reduce 5 commands to 3 (init, sync, edit). Add iA Writer integration for external editing with sync-back. Enforce a hard < 150 line budget with pre-commit hook enforcement.

    Primary User Story¶

    As a developer using Claude Code, I want my CLAUDE.md to be as lean as possible (< 150 lines), so that every session starts with minimal token overhead while still having access to full project documentation on-demand via pointers.

    Acceptance Criteria¶

    • [ ] CLAUDE.md files generated by init are < 150 lines
    • [ ] sync command refuses to add release notes (hard block)
    • [ ] sync --optimize moves bloated sections to detail files
    • [ ] edit command opens iA Writer with TODO annotations
    • [ ] Pre-commit hook blocks commits where CLAUDE.md > 150 lines
    • [ ] Pointer format: -> Full history: [VERSION-HISTORY.md](docs/VERSION-HISTORY.md)
    • [ ] All 4 standard detail files supported (VERSION-HISTORY, ARCHITECTURE, CONTRIBUTING, COMMANDS)
    • [ ] --global flag works on ~/.claude/CLAUDE.md

    Secondary User Stories¶

    As a craft plugin maintainer, I want the sync command to auto-audit after updating metrics, so that I catch issues in a single command invocation.

    As a developer working across multiple projects, I want the global CLAUDE.md to use pointers to project-specific docs, so that project-specific detail doesn't pollute every session.

    Architecture¶

    Command Refactoring: 5 -> 3¶

    BEFORE (v2.10):                    AFTER (v3.0):
commands/docs/claude-md/           commands/docs/claude-md/
├── scaffold.md  ─────────────>    ├── init.md      (scaffold + first setup)
├── update.md    ──┐               ├── sync.md      (update + audit + fix + optimize)
├── audit.md     ──┤──────────>    └── edit.md      (iA Writer integration)
├── fix.md       ──┘
└── edit.md      ─────────────>

    Command Mapping¶

    Old Command New Command Behavior Change
    scaffold init Same core logic. Refuses if CLAUDE.md exists (unless --force). Uses lean templates (< 150 lines).
    update sync Updates metrics + always audits + reports issues. --fix auto-fixes. --optimize enforces budget.
    audit sync (embedded) Audit runs automatically after every sync. No standalone command.
    fix sync --fix Auto-fix is a flag on sync, not a separate command.
    edit edit Enhanced: opens iA Writer via AppleScript, adds TODO hints, waits for 'done', re-reads + validates.

    Flags¶

    /craft:docs:claude-md:init¶

    Flag Short Default Description
    --type -t auto-detect Template type: plugin, teaching, r-package
    --force -f false Overwrite existing CLAUDE.md
    --dry-run -n false Preview without creating
    --global -g false Target ~/.claude/CLAUDE.md

    /craft:docs:claude-md:sync¶

    Flag Short Default Description
    --fix false Auto-fix issues found during audit
    --optimize -o false Enforce < 150 line budget, move bloat to detail files
    --dry-run -n false Preview changes without applying
    --global -g false Target ~/.claude/CLAUDE.md
    --section -s all Specific section: status, commands, architecture

    /craft:docs:claude-md:edit¶

    Flag Short Default Description
    --hints true Add TODO optimization comments before opening
    --no-hints false Open without annotations
    --global -g false Target ~/.claude/CLAUDE.md
    --validate true Run audit after edit completes

    Pointer Architecture¶

    Format Standard¶

    Pointer lines use markdown link with arrow prefix:

    -> Full release history: [VERSION-HISTORY.md](docs/VERSION-HISTORY.md)

    Claude Code naturally understands markdown links and will Read the target file when the user asks about that topic. The arrow prefix (->) makes pointers visually distinct from regular links.

    Standard Detail Files¶

    Detail File Contains Pointer In CLAUDE.md
    docs/VERSION-HISTORY.md All release notes, changelogs, migration guides -> Release history: VERSION-HISTORY.md
    docs/ARCHITECTURE.md Detailed system design, component diagrams, data flows -> Architecture detail: ARCHITECTURE.md
    CONTRIBUTING.md Dev setup, testing guide, PR process details -> Contributing guide: CONTRIBUTING.md
    docs/COMMANDS.md Full command/API reference with all options and examples -> Command reference: COMMANDS.md

    P0 Sections (Always in CLAUDE.md)¶

    These sections are never cut, regardless of budget:

    Section Max Lines Content
    Header/TL;DR 5 Name, version, one-liner, key stats
    Git Workflow 15 Branch strategy, commit conventions, worktree commands
    Quick Commands 15 Build, test, lint, validate — the daily-use commands
    Project Structure 15 Directory layout with key files (tree format)
    Troubleshooting 10 Top 5-8 most common issues and fixes
    Pointers 10 Links to detail files
    Total P0 ~70 Leaves 80 lines for P1 content

    P1 Sections (Include if under budget)¶

    Section Max Lines Content
    Agents/Routing 10 Agent table for task routing
    Execution Modes 8 Mode table (default, debug, optimize, release)
    Active Development 5 Current branch + version only
    Key Files 15 Top 15 most-referenced files

    P2 Sections (Always in detail files, never in CLAUDE.md)¶

    Content Detail File
    Release notes/history VERSION-HISTORY.md
    Feature matrices VERSION-HISTORY.md
    Per-test-file breakdowns CONTRIBUTING.md
    Phase completion details VERSION-HISTORY.md
    PR/merge details VERSION-HISTORY.md
    Diffstats (+lines/-lines) Never documented
    Implementation phase notes Never documented (delete after release)

    Anti-Pattern Rules (Hard Blocks)¶

    The sync command refuses to add these to CLAUDE.md:

    Pattern Detection Action
    Release notes sections Regex: ^###? v\d+\.\d+ or Released \d{4} Block + redirect to VERSION-HISTORY.md
    Diffstats Regex: Files Changed:.*\+.*/- Block + delete
    "What Shipped" sections Regex: What Shipped\|Merged PR Block + redirect
    Completed feature tables Regex: Status.*Complete\|Released ✅ Block + redirect
    Phase implementation details Regex: ^### Phase \d with completion markers Block + redirect

    TODO Hint Format¶

    Before opening, Claude inserts HTML comments with optimization guidance:

    <!-- TODO: Section "Recent Major Features" is 174 lines. Target: 0 lines.
     Action: Move to docs/VERSION-HISTORY.md, replace with pointer. -->
## Recent Major Features

<!-- TODO: Section "Test Suite" is 25 lines. Target: 8 lines.
     Action: Keep run command only, remove per-file breakdown. -->
## Test Suite

    Pre-Commit Hook¶

    Hook Configuration¶

    Add to .pre-commit-config.yaml:

    - repo: local
  hooks:
    - id: claude-md-budget
      name: CLAUDE.md line budget
      entry: bash -c 'lines=$(wc -l < CLAUDE.md 2>/dev/null || echo 0); if [ "$lines" -gt 150 ]; then echo "CLAUDE.md is $lines lines (budget: 150). Run /craft:docs:claude-md:sync --optimize"; exit 1; fi'
      language: system
      files: CLAUDE.md
      pass_filenames: false

    CI Integration¶

    Add to docs-quality.yml:

    - name: Check CLAUDE.md budget
  run: |
    lines=$(wc -l < CLAUDE.md)
    if [ "$lines" -gt 150 ]; then
      echo "::error::CLAUDE.md is $lines lines (budget: 150)"
      exit 1
    fi

## Migration Plan

### Phase 1: Utility Refactoring

1. Create `claude_md_sync.py` merging auditor + fixer + updater logic
2. Create `claude_md_optimizer.py` with budget enforcement
3. Add pointer generation to optimizer
4. Add anti-pattern detection (release notes, diffstats)
5. Tests for new modules

### Phase 2: Command Refactoring

1. Create `init.md` (from scaffold.md with lean templates)
2. Create `sync.md` (from update.md with embedded audit/fix/optimize)
3. Update `edit.md` (add iA Writer workflow + TODO hints)
4. Remove old command files (update.md, audit.md, fix.md, scaffold.md)
5. Update hub.md references

### Phase 3: Template & Documentation

1. Create lean template variants (< 150 lines)
2. Add pre-commit hook for budget enforcement
3. Update all documentation references (REFCARD, tutorials, etc.)
4. Update CLAUDE.md itself to use the new lean format
5. Update test suite

### Phase 4: Global CLAUDE.md

1. Add --global flag to all 3 commands
2. Create pointer pattern for project-specific sections
3. Apply lean optimization to ~/.claude/CLAUDE.md

## Resolved Questions

| # | Question | Decision | Rationale |
|---|----------|----------|-----------|
| 1 | Backward compatibility? | **1 release of aliases** | v2.12 adds new + deprecation warnings. v2.13 removes old names. |
| 2 | Auto-create detail files? | **Yes, create with moved content** | If docs/VERSION-HISTORY.md doesn't exist, create it and move content there. |

## Git Workflow

`main` (protected, PR only) <- `dev` (planning, no feature code) <- `feature/*` (worktrees)

| Step | Action | Command |
|------|--------|---------|
| Plan | Analyze on `dev`, wait for approval | `git checkout dev` |
| Branch | Create worktree | `/craft:git:worktree feature/<name>` |
| Develop | Conventional commits | `feat:`, `fix:`, `docs:` |
| Integrate | PR to dev | `gh pr create --base dev` |
| Release | PR to main | `gh pr create --base main --head dev` |

**CRITICAL**: Never commit to `main`. Never write feature code on `dev`. Always verify: `git branch --show-current`
