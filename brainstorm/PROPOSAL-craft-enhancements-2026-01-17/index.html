    Craft Command Enhancements - Claude Code 2.1.0 Integration¶

    Generated: 2026-01-17 Context: Leverage Claude Code 2.1.0 features for /craft:do, /craft:orchestrate, /craft:check Research: Based on Claude Code 2.1.0 release (January 7, 2026)

    Executive Summary¶

    Claude Code 2.1.0 introduces powerful agent and skill features that can significantly enhance Craft's command capabilities:

    • Agent resilience - Subagents continue after permission denial
    • Skills hot-reload - Automatic detection without restart
    • Forked context - Skills can execute in isolated context
    • Agent hooks - PreToolUse, PostToolUse, Stop scoped to agent lifecycle
    • Session teleportation - /teleport to claude.ai/code
    • Language support - Configure response language

    This proposal outlines how to integrate these features into three priority commands.

    Current State Analysis¶

    /craft:do (Smart Routing)¶

    Current Implementation: - Simple keyword-based routing - No agent delegation - Routes to other craft commands - Supports dry-run mode - Checks for existing specs

    Architecture: 

    User Task ‚Üí Keyword Analysis ‚Üí Route to Commands
                ‚Üì
         [No Agent Usage]

    Limitations: - Cannot handle complex multi-step tasks - No parallel execution - Limited to predefined routing rules - Cannot adapt to task complexity

    /craft:orchestrate (Multi-Agent Coordination)¶

    Current Implementation: - Uses orchestrator-v2 agent - Parallel and sequential wave execution - Resource monitoring (max 2 concurrent agents) - Chat compression at 70% context - Dry-run mode

    Architecture: 

    User Task ‚Üí orchestrator-v2 Agent
                ‚Üì
         Wave Planning
                ‚Üì
     [Agent 1] [Agent 2] (parallel)
                ‚Üì
         [Agent 3] (sequential)

    Limitations: - Agent failures can block entire wave - No forked context isolation - Limited error recovery - No agent hook integration

    /craft:check (Validation)¶

    Current Implementation: - Pre-flight validation for commits/releases - Static validation rules - No dynamic checks

    Limitations: - No hot-reload of new validation rules - Cannot leverage agent-based validation - Limited extensibility

    Enhancement Proposals¶

    üéØ Enhancement 2: /craft:orchestrate with Agent Hooks¶

    Overview¶

    Integrate Claude Code 2.1.0 agent hooks for better monitoring, error recovery, and resource management.

    Architecture Enhancement¶

    orchestrator-v2 Agent

    Implementation Strategy¶

    Phase 1: Agent Hook Integration (v1.23.0)¶

    Create .claude-plugin/hooks/orchestrate-hooks.sh:

    #!/bin/bash
# Agent-scoped hooks for orchestrator-v2

case "$HOOK_EVENT" in
  PreToolUse)
    # Check resource availability before agent launch
    ACTIVE_AGENTS=$(pgrep -f "claude.*agent" | wc -l)
    if [ "$ACTIVE_AGENTS" -ge 2 ]; then
      echo "‚ö† Resource limit: 2 agents already running"
      echo "action: queue"
      exit 0
    fi

    # Log agent start
    echo "[$(date)] Starting agent: $TOOL_NAME" >> .craft/logs/orchestration.log
    ;;

  PostToolUse)
    # Log agent completion
    echo "[$(date)] Completed agent: $TOOL_NAME (duration: ${DURATION}s)" >> .craft/logs/orchestration.log

    # Update session cache
    echo "$TOOL_NAME:success:$(date +%s)" >> .craft/cache/agent-results.cache
    ;;

  Stop)
    # Cleanup on orchestration stop
    echo "[$(date)] Orchestration stopped" >> .craft/logs/orchestration.log

    # Save session state
    jq -n \
      --arg timestamp "$(date -Iseconds)" \
      --arg agents "$AGENT_COUNT" \
      '{timestamp: $timestamp, total_agents: $agents}' \
      > .craft/cache/last-orchestration.json
    ;;
esac

    Hook Configuration:

    {
  "hooks": {
    "PreToolUse": {
      "command": "/bin/bash .claude-plugin/hooks/orchestrate-hooks.sh",
      "scope": "agent",
      "agents": ["orchestrator-v2"]
    },
    "PostToolUse": {
      "command": "/bin/bash .claude-plugin/hooks/orchestrate-hooks.sh",
      "scope": "agent",
      "agents": ["orchestrator-v2"]
    },
    "Stop": {
      "command": "/bin/bash .claude-plugin/hooks/orchestrate-hooks.sh",

    }
  }
}

    Phase 2: Agent Resilience (v1.24.0)¶

    Leverage subagent resilience (agents continue after permission denial):

    Before (Current): 

    Wave 1: [Agent A] [Agent B]
         ‚Üì denied    ‚Üì success

    After (Resilient): 

    Wave 1: [Agent A] [Agent B]
         ‚Üì denied    ‚Üì success
         ‚Üì continue

         ‚Üì

    Implementation in orchestrate.md:

    ## Wave Planning (Resilient Mode)

Waves proceed even if some agents are denied:

1. **Collect available results** from successful agents
2. **Mark denied agents** for manual review
3. **Continue with partial results** if dependencies satisfied
4. **Report missing inputs** in final synthesis

### Example:
    Wave 1 (Parallel - 2 agents): ‚îú‚îÄ Agent: arch-1 (architecture) ‚Üí DENIED (user skipped) ‚îî‚îÄ Agent: doc-1 (documentation) ‚Üí SUCCESS

    Wave 2 (Sequential): ‚îú‚îÄ Agent: code-1 (backend) ‚îÇ Dependencies: arch-1 (MISSING - will use fallback) ‚îÇ Fallback: Use general best practices ‚îî‚îÄ Agent: test-1 (testing) Dependencies: code-1 

    **Benefits:**
- Orchestration doesn't fail completely
- User can skip non-critical agents
- Graceful degradation with fallbacks

    Recommended Implementation Path¶

    Priority 1: /craft:do Agent Delegation (Weeks 1-2)¶

    Why first: - Biggest user-facing impact - Leverages most Claude Code 2.1.0 features - Enables complex task handling

    Phases: 1. Complexity scoring (1-2 hours) 2. Agent routing (2-3 hours) 3. Forked context (1-2 hours) 4. Testing and dry-run (2-3 hours)

    Total effort: ~8-10 hours

    Priority 2: /craft:check Hot-Reload (Weeks 2-3)¶

    Why second: - Quick wins available - Community extensibility - Low risk, high value

    Phases: 1. Create 3-5 validation skills (2-3 hours) 2. Build discovery system (2-3 hours) 3. Test hot-reload (1 hour) 4. Documentation (1-2 hours)

    Total effort: ~6-9 hours

    Priority 3: /craft:orchestrate Agent Hooks (Weeks 3-4)¶

    Why third: - Builds on existing orchestration - Requires agent delegation from Priority 1 - More complex integration

    Phases: 1. Create hook scripts (1-2 hours) 2. Integrate resilience (3-4 hours) 3. Add forked context (2-3 hours) 4. Enhanced monitoring (2-3 hours)

    Total effort: ~8-12 hours
